## 6. Projets à créer

L’objectif n’est pas d’accumuler des démonstrations. Chaque projet doit combler une faiblesse précise du portfolio et fournir une preuve publique absente des projets professionnels.

### Priorité 1 - Produit SaaS métier déployé dans le cloud

Créer une application complète autour d’un besoin métier réaliste, sans reproduire une solution confidentielle.

**Preuves attendues :**

- backend C#/.NET et API REST documentée ;
- interface Angular ou Blazor ;
- base de données PostgreSQL ou MySQL ;
- authentification et gestion des autorisations ;
- tâche planifiée et envoi de rapports par e-mail ;
- tests automatisés ;
- pipeline CI/CD ;
- déploiement sur Azure ou un autre cloud ;
- supervision, journalisation et documentation d’exploitation.

**Manques couverts :** code public full-stack, cloud, tests, déploiement et livraison d’un produit complet pour un client.

### Priorité 2 - Application full-stack .NET et React

Créer une application plus petite, centrée sur la qualité de l’expérience utilisateur et la consommation d’une API.

**Preuves attendues :**

- frontend React moderne, responsive et accessible ;
- backend .NET ;
- gestion d’état, formulaires et erreurs ;
- tests frontend ;
- démonstration publique et dépôt GitHub documenté.

**Manque couvert :** faire passer React et JavaScript du niveau « en progression » à un niveau démontrable.

### Priorité 3 - Outil .NET open source

Créer une bibliothèque, un package NuGet ou un outil en ligne de commande répondant à un problème technique limité et réel.

**Preuves attendues :**

- API claire et versionnée ;
- tests unitaires et d’intégration ;
- documentation d’installation et exemples ;
- automatisation GitHub Actions ;
- publication d’une version et gestion des changements.

**Manques couverts :** code publiquement inspectable, tests, documentation technique et contribution open source.

### Portfolio narratif

Le portfolio à la première personne, mêlant horreur légère et humour, constitue un projet distinctif de présentation. Il démontrera la créativité et l’attention portée à l’expérience utilisateur, mais ne devra pas remplacer les preuves techniques apportées par les projets ci-dessus.

## 11. Règles éditoriales

- Ne jamais inventer de métrique, de responsabilité ou de résultat.
- Ne pas publier de code, capture, architecture interne ou donnée confidentielle provenant d’un employeur ou d’un client.
- Ne pas confondre utilisation d’une architecture existante et responsabilité de sa conception.
- Ne pas confondre Azure DevOps et expérience du cloud Azure.
- Présenter les projets professionnels comme des études de cas fondées sur le contexte, les actions personnelles et les résultats vérifiables.
- Ajouter une compétence uniquement lorsqu’une expérience professionnelle ou un projet public permet de la justifier.
